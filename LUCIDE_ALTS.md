# Architecture Alternatives — @finografic/lucide-manager

Captured after the initial integration with `@workspace/design-system`.
Current state, problems encountered, and every alternative discussed — for picking back up later.

---

## Current Architecture (as of 0.3.0)

```
@finografic/lucide-manager (published npm package)
  bin/lucide-manager.js     ← CLI shim
  src/
    config/loadConfig.ts    ← walks up from cwd to find lucide-manager.config.json
    server/plugin.ts        ← Vite middleware for GET/POST /api/icons-json
  scripts/
    generate-icons-ts.ts    ← reads icons.json → writes icons.ts + index.ts
  vite.config.ts            ← loads the plugin, serves the React picker UI

Host (packages/design-system/)
  lucide-manager.config.json        ← { iconsJsonPath, iconsDir }
  src/icons/
    icons.json              ← source of truth (picker reads/writes this)
    icons.ts                ← generated
    index.ts                ← generated
    icons.utils.ts          ← createIconWrapper (hand-written, never overwritten)
```

**Commands:**

- `lucide-manager dev` → spawns Vite at `:5199` with `LUCIDE_MANAGER_HOST_CWD` env var so `loadConfig()` finds the host's config despite Vite running with `cwd = pkgRoot`
- `lucide-manager generate` → runs `generate-icons-ts.ts` via `tsx`

**Pain points:**

- `vite` and `tsx` must be `dependencies` (not `devDependencies`) to survive pnpm's install model
- `findBin()` in the CLI shim searches 5 candidate paths to locate the binaries — fragile across package managers
- The `LUCIDE_MANAGER_HOST_CWD` env var trick is necessary but non-obvious
- Self-dev mode detection (`cwd === PKG_ROOT`) is a footgun
- Generated `index.ts` had a TS bug (re-export ≠ local binding) that required a patch release

## Alternative A — Pre-built Static Bundle (Zero Runtime Deps)

**Concept:** Build the React picker UI once at publish time. Ship `dist/` in the package. Replace the Vite dev server with a tiny Node `http.createServer()` inside the bin itself.

**Structure:**

```
@finografic/lucide-manager
  bin/lucide-manager.js   ← starts Node http server (no spawn, no findBin)
  dist/                   ← pre-built picker UI (committed or built in CI)
  scripts/
    generate-icons-ts.ts
```

**How `dev` works:**

```
lucide-manager dev
  → reads lucide-manager.config.json directly (no cwd tricks)
  → starts http.createServer() on :5199
  → serves dist/ as static files
  → handles /api/icons-json GET/POST inline
```

**Key changes vs. current:**

- `vite`, `react`, `tsx` → all `devDependencies` (build-time only)
- Zero runtime `dependencies`
- No `findBin()`, no `LUCIDE_MANAGER_HOST_CWD`, no self-dev mode
- Instant startup (no Vite cold start)

**Tradeoffs:**

- Picker UI changes require a publish (no hot-reload while developing the picker itself — only matters when working on lucide-manager, not when consuming it)
- Must build `dist/` before publishing (CI step or committed artifact)

**Verdict:** Best DX for consumers. Most work to refactor.

## Alternative B — Decouple Codegen into the DS

**Concept:** Move `generate-icons-ts.ts` into `packages/design-system/scripts/`. The DS owns its own codegen. `@finografic/lucide-manager` becomes purely a picker UI with no knowledge of file paths or generation.

**Structure:**

```
packages/design-system/
  scripts/
    generate-icons-ts.ts   ← moved here, runs as local tsx script
    icons-server.ts        ← tiny Node http server (new)
  src/icons/
    icons.json
    icons.ts               ← generated by local script
    index.ts               ← generated by local script
  package.json
    "icons.dev":      "tsx scripts/icons-server.ts"
    "icons.generate": "tsx scripts/generate-icons-ts.ts"
```

**How `dev` works:**

```
pnpm icons.dev (in DS)
  → DS starts its own http server
  → serves pre-built picker UI from node_modules/@finografic/lucide-manager/dist/
  → handles /api/icons-json, writes icons.json, triggers generate on save
```

**Key changes vs. current:**

- No CLI bin needed in lucide-manager for the generate command
- The DS is fully self-contained for the codegen pipeline
- lucide-manager is only responsible for the picker UI artifact

**Tradeoffs:**

- DS gains a dev-only server script (small, but it's a new moving part)
- Two repos to update when the API contract changes (lucide-manager UI format ↔ DS server)
- More setup for new consumers (must wire up server script themselves)

**Verdict:** Good if codegen needs to be customised per-project. Overkill if the generate logic stays stable.

## Alternative A + B Hybrid

**Concept:** Combine A (pre-built UI, no Vite runtime) with B (codegen in DS). lucide-manager becomes a pure UI artifact with zero moving parts. The DS owns everything operational.

**Responsibility split:**

| What                                | Lives in                                                          |
| ----------------------------------- | ----------------------------------------------------------------- |
| Picker React UI (visual)            | `@finografic/lucide-manager` — pre-built `dist/`                  |
| File server + `/api/icons-json`     | `packages/design-system/scripts/icons-server.ts` (Node built-ins) |
| Codegen (`icons.json` → `icons.ts`) | `packages/design-system/scripts/generate-icons-ts.ts`             |
| Source of truth                     | `packages/design-system/src/icons/icons.json`                     |

**Why it's clean:** lucide-manager has no server, no file access, no config walking. It's just a compiled UI. The DS controls the full dev loop. Bumping the picker version is just `pnpm update @finografic/lucide-manager`.

**Why it may not be worth it:** The DS now needs to run a server script. That's a new thing to explain, maintain, and wire up in other projects. The current approach keeps that complexity inside the CLI package where consumers don't see it.

**Verdict:** Cleanest architecture but the DS gains operational complexity. Best if the DS is the only consumer and you control both repos.

## Alternative C — VS Code Extension with WebView

**Concept:** Ship the icon picker as a VS Code extension. Picker opens as a WebView panel. File reads/writes go through the VS Code `workspace.fs` API — no HTTP server, no CLI, no dep resolution.

**How it works:**

- Command Palette: `Lucide Manager: Open Icon Picker`
- Extension finds `lucide-manager.config.json` in the workspace root
- Opens a WebView panel with the compiled React picker UI (loaded from `extension/media/`)
- On save: extension host writes `icons.json` and optionally triggers the generate script via `vscode.tasks.executeTask`

**Tradeoffs:**

- No server, no Vite at runtime, no dep resolution — zero moving parts at runtime
- Deep editor integration (can open files, jump to usages, etc.)
- VS Code only — no terminal-only or CI usage
- Extension packaging + Marketplace publishing is its own process
- This is where the ecosystem is heading — Panda Studio's VS Code extension, Tailwind IntelliSense, etc.

**Note on Panda Studio:** Panda Studio uses Vite (`vite.createServer()` is a direct dep of `@pandacss/studio`). If you add Panda Studio to the DS, Vite will already be in the monorepo — which makes the current approach more reliable and makes Alternative A (build-time only Vite) even more attractive.

**Verdict:** Best long-term if you're committed to VS Code. Most work to build initially.

## Alternative D — `packages/icons/` as a Dedicated Workspace Package

**Concept:** Extract icons out of the DS entirely into a new first-class monorepo package. Icons are no longer a subfolder — they are an independently versioned, publishable package.

**Structure:**

```
packages/icons/
  src/
    icons.json              ← source of truth
    icons.ts                ← generated
    index.ts                ← generated
    icons.utils.ts          ← createIconWrapper
  lucide-manager.config.json
  package.json              ← name: "@workspace/icons" / published as "@finografic/icons"
```

**Import paths:**

```ts
// Inside monorepo — direct, no build needed
import { ChevronDownIcon } from '@workspace/icons';

// Other projects — installed from registry
import { ChevronDownIcon } from '@finografic/icons';
```

**DS relationship:** The DS imports from `@workspace/icons` directly (or re-exports for backwards compat).

**Two usage modes for other projects:**

1. Install `@finografic/icons` — get the pre-built registry as-is
2. Run `@finografic/lucide-manager` in their own project — generate a custom icon set in their own `packages/icons/`

**Advantages over current:**

- Icons versioned independently of the DS — icon changes don't require a DS release
- Faster turborepo builds — icon changes don't invalidate the DS cache
- Other packages in the monorepo can depend on `@workspace/icons` without pulling in all of the DS
- Pattern is reusable — any new monorepo can adopt `@finografic/icons` or run lucide-manager to build their own

**Honest counter:** If icons always change in lockstep with the DS and nothing else needs icons independently, this is overhead. Do it when you actually feel the friction — not before.

**Verdict:** Best long-term scalability. Lowest urgency right now.

## Summary

|             | Runtime deps | Server complexity     | Portable          | Best for                     |
| ----------- | ------------ | --------------------- | ----------------- | ---------------------------- |
| **Current** | vite + tsx   | Inside lucide-manager | Yes (CLI)         | Now — works, ships           |
| **Alt A**   | None         | Inside lucide-manager | Yes (CLI)         | Cleaner version of current   |
| **Alt B**   | None         | Inside DS             | No (DS-specific)  | DS controls everything       |
| **Alt A+B** | None         | Inside DS             | No (DS-specific)  | Cleanest arch, most DS work  |
| **Alt C**   | None         | None (VS Code API)    | VS Code only      | Long-term editor integration |
| **Alt D**   | Unchanged    | Unchanged             | Yes (publishable) | When icons outgrow the DS    |

**Most likely next step:** Alt A (pre-built bundle) is a clean, contained improvement to the current approach with no architectural changes. Alt D (`packages/icons/`) is worth doing if a second project needs icons independently.
